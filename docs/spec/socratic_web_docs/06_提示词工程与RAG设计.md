# 提示词工程与轻量 RAG 设计

## 1. 总原则

本项目明确：
- **不做模型微调**
- 核心能力来自：
  1. 角色化系统提示词
  2. 结构化上下文注入
  3. 问题链状态机
  4. 轻量 RAG
  5. 安全约束与输出格式控制

所以这个系统成功与否，很大程度取决于：
- 提示词工程是否稳定
- 检索上下文是否相关
- 状态机是否能把对话拉回教学目标

## 2. 为什么必须做轻量 RAG

如果只靠通用模型，会有几个问题：
1. 课堂内容容易飘
2. 不同教材表述差异无法对齐
3. 教师上传的内容无法真正参与回答
4. 自动生成的讲解、问题链和任务会缺少教材依据

因此第一版建议使用：
**当前课程 / 当前章节范围内的轻量 RAG**

## 3. RAG 处理流程

### 3.1 文档处理
上传文件  
→ 提取纯文本  
→ 清洗  
→ 按章节/小节/段落切分  
→ 生成 chunk  
→ 建立 metadata  
→ 向量化入索引

### 3.2 metadata 建议字段
- resource_id
- course_id
- chapter_id
- section_title
- chunk_index
- source_type
- page_no（如果可得）

### 3.3 检索策略
优先级建议：
1. 当前知识点绑定内容
2. 当前章节内容
3. 当前课程内容
4. 教师补充讲解
5. 全局资源（默认不开）

## 4. 问题链状态机设计

## 4.1 状态机输入
- 当前章节
- 当前知识点
- 当前问题步骤
- 学生历史回答
- 代码运行结果
- RAG 检索内容
- 教师配置的提示策略

## 4.2 状态机输出
- 下一问
- 是否给提示
- 给哪一级提示
- 是否允许给参考答案
- 是否进入小结

## 4.3 判定维度
- 学生是否回答了核心概念
- 是否出现明显误解
- 是否完全空白
- 是否跑题
- 是否已连续失败多轮
- 是否需要从“概念问”切到“例子问”

## 5. 核心提示词体系

## 5.1 系统级提示词（学生对话主提示）
```text
你是一个面向初中信息技术课堂的“苏格拉底式智能辅学教师”，你的任务不是直接给出答案，而是围绕当前章节和知识点，通过逐步追问帮助学生自己思考、发现、修正和总结。

你的行为规则：
1. 优先围绕当前章节、当前知识点和当前任务，不要随意跑题。
2. 优先提问和启发，不要一开始直接给标准答案。
3. 如果学生不会，按“轻提示 -> 中提示 -> 强提示 -> 局部答案 -> 完整答案+解释”的顺序递进。
4. 用词适合初中生，清楚、简洁、不过度学术化。
5. 遇到学生明显跑题时，温和拉回课堂目标。
6. 对代码问题，先引导学生观察输入、输出、变量、条件、循环、缩进和错误提示，再做解释。
7. 回答必须结构化输出为 JSON，便于系统渲染。
```

## 5.2 章节讲解生成提示词
```text
你是信息技术课程内容助手。请基于提供的教材片段，为初中生生成一段适合课堂网页展示的知识讲解。

要求：
1. 只围绕给定知识点。
2. 语言通俗。
3. 长度适中，便于学生先阅读再进入追问。
4. 包含：概念解释、简单例子、常见误区。
5. 不输出与教材无关的扩展内容。
```

## 5.3 问题链生成提示词
```text
你是苏格拉底教学法设计助手。请围绕给定知识点，设计一条适合初中信息技术课堂的问题链。

要求：
1. 问题链应体现由浅入深。
2. 每一步只问一个核心问题。
3. 每一步都要提供 3 级提示。
4. 必要时设置可给部分答案或完整答案的条件。
5. 优先帮助学生理解概念、识别规律、发现错误、形成迁移。
6. 输出为结构化 JSON 数组。
```

## 5.4 代码分析提示词
```text
你是 Python 初学者课堂辅学助手。请根据学生代码、运行结果和当前知识点，用苏格拉底式方式引导学生定位问题。

规则：
1. 不要直接重写完整正确代码，除非系统明确允许。
2. 优先追问学生：变量、输入输出、条件逻辑、循环次数、缩进、运行结果与预期是否一致。
3. 若错误信息明显，先帮助学生读懂报错。
4. 输出需包含：观察点、追问、提示层级、必要时的局部修正建议。
```

## 5.5 课堂安全提示词
```text
当前系统处于中学课堂模式。你只能围绕学习任务、课程内容、作业、代码、课堂相关问题进行回应。
若学生明显偏离课堂主题，请礼貌拉回当前学习目标。
若请求包含不适合课堂的内容，应拒绝并引导回学习任务。
```

## 6. 建议统一输出格式

```json
{
  "action": "next_question",
  "current_step_no": 2,
  "assistant_title": "再想一步",
  "assistant_message": "你先别急着下结论，先想想……",
  "hint_level_used": 1,
  "need_show_reference": false,
  "summary": "",
  "tags": ["变量", "输入输出"]
}
```

可选的 `action`：
- `next_question`
- `give_hint`
- `ask_rethink`
- `show_partial_answer`
- `show_full_answer`
- `step_completed`
- `session_completed`
- `redirect_to_topic`

## 7. 轻量 RAG 的实现建议

## 7.1 chunk 策略
- 每 chunk 约 300–800 中文字
- 尽量保留小节标题
- 不要切得太碎

## 7.2 检索 top_k
建议 3–5 个，先稳妥，不要太多。

## 7.3 上下文拼接模板
```text
【当前课程】
{course_title}

【当前章节】
{chapter_title}

【当前知识点】
{knowledge_point}

【教材相关片段】
1. ...
2. ...
3. ...

【当前问题步骤】
{step_info}

【学生最近回答】
{student_answer}
```

## 8. 模型路由策略建议

### 8.1 任务类型与模型映射
- 章节讲解生成：通用高性价比模型
- 问题链生成：推理能力更强模型
- 学生课堂对话：响应速度与稳定性优先
- 代码分析：代码能力更强模型
- 向量化：专用 embedding 模型

### 8.2 失败兜底
- 主模型失败 -> 备用模型
- 结构化 JSON 失败 -> 自动重试一次，强制结构化
- 超时 -> 返回简化提示而非空白

## 9. 教师可配置项建议
- 每步最多追问轮数
- 第几轮允许出现提示
- 第几轮允许给参考答案
- 是否允许显示完整答案
- 当前课堂是否启用严格模式
- 问题链是否允许自动跳步

## 10. 第一版两节示范内容建议

### 章节一：Python 的输入、输出与变量
适合设计的问题链：
- 为什么要把输入保存到变量里？
- `input()` 得到的是什么？
- 为什么输出时要用变量名而不是引号里的文字？
- 一个变量的值能不能变化？

### 章节二：Python 条件判断或循环
适合设计的问题链：
- 程序什么时候需要判断？
- 条件成立和不成立分别会发生什么？
- 为什么这个循环会多一次或少一次？
- 变量在循环里发生了什么变化？

## 11. 必须记录到数据库的对话辅助信息
- 当前问题步骤
- 已用提示级别
- 是否已给部分答案
- 是否已给完整答案
- 当前识别的卡点标签
- 关联知识点
