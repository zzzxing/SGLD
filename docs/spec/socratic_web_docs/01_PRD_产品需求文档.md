# PRD｜苏格拉底式智能辅学 Web 系统（追问智学）

## 1. 产品背景

传统信息技术课堂里，学生常见问题不是“没有内容”，而是：

1. 知识听过但不会迁移
2. 任务会抄不会想
3. 课堂里教师无法同时对全班每个人持续追问
4. 学生代码或操作出错时，只能得到结果性纠错，缺乏过程性启发
5. 教师很难实时掌握全班卡点

本系统的目标，是把“苏格拉底式追问”做成一个可运行的课堂软件，而不是一个泛对话 AI。

## 2. 产品目标

### 2.1 总目标
构建一个在机房局域网内运行的 Web 教学平台，使教师能够组织全班围绕某一章节开展**知识讲解—问题链引导—任务实践—过程记录—课堂统计**的完整学习流程。

### 2.2 第一版目标
- 让学生在网页中围绕章节内容进行受控学习
- 让教师能低成本配置教学资源和问题链
- 让管理员能管理账号、模型 API、资源库和系统参数
- 让系统能在不微调模型的情况下，依靠提示词工程和轻量 RAG 形成可用教学体验

## 3. 用户角色与核心诉求

## 3.1 学生
### 诉求
- 知道今天学什么
- 看得懂当前知识点
- 能被一步步追问，不是直接被给答案
- 能提交文字、代码、作业结果
- 能获得引导、提示、纠错和小结

### 痛点
- 容易跑题
- 不知道自己卡在哪里
- 直接问 AI 容易得到答案，但不会真正理解

## 3.2 教师
### 诉求
- 尽量少配置复杂细节
- 能上传教材，系统自动生成章节、知识点、讲解和问题链初稿
- 能布置全班进入同一章节/任务
- 能实时看到谁卡住了、谁完成了
- 能回看学生表现并做简单评价

## 3.3 管理员
### 诉求
- 管账号、班级、教师、学生
- 管模型 API 和安全参数
- 管资源库
- 管系统级设置与运行状态

## 4. 使用场景

## 4.1 典型课堂场景
1. 教师登录教师端
2. 选择班级、选择章节、点击“开始上课”
3. 系统通过实时推送，让全班学生端自动进入本节学习页
4. 学生阅读本节知识讲解
5. 系统根据问题链逐步追问
6. 学生提交文字思路 / Python 代码 / 任务结果
7. 系统运行代码并结合模型进行追问、纠错、总结
8. 教师在大屏或教师端看到全班完成率、当前进度和卡点分布
9. 教师可对个别学生发点评或统一补充说明
10. 课堂结束后沉淀过程记录和统计数据

## 4.2 教材配置场景
1. 教师上传 Word/PDF/PPT/文本
2. 系统自动解析为：章节—小节—知识点
3. 系统生成：讲解初稿、问题链初稿、任务初稿
4. 教师进入编辑页做精修
5. 保存后发布给班级

## 5. 核心设计原则

1. **课堂优先，不做开放闲聊产品**
2. **学生端聚焦，不让界面过重**
3. **教师低负担，系统多自动生成**
4. **管理员克制但必要**
5. **先做能跑的闭环，再做高级功能**
6. **UI 要现代 AI 产品风，但功能逻辑要教育化**
7. **所有“智能”能力都必须可被教师和系统边界约束**

## 7. 系统功能清单

## 7.1 管理员端
1. 管理教师账号
2. 管理班级
3. 管理学生账号
4. 管理模型供应商与 API Key
5. 管理模型可见范围
6. 管理资源库
7. 管理系统配置
8. 查看系统运行日志、调用统计、错误日志
9. 管理安全策略

## 7.2 教师端
1. 查看班级
2. 上传教材/讲义
3. 编辑章节树
4. 编辑知识点
5. 编辑讲解内容
6. 查看/修改系统生成的问题链
7. 查看/修改任务与练习
8. 发布章节或任务给班级
9. 一键开始课堂
10. 实时查看全班进度
11. 查看学生对话和结果
12. 点评/打标/评分

## 7.3 学生端
1. 登录
2. 进入当前班级课堂
3. 查看章节知识讲解
4. 按问题链作答
5. 输入文字思路
6. 输入或粘贴 Python 代码
7. 运行代码并查看结果
8. 上传任务结果文件
9. 接受系统提示与追问
10. 查看当前学习进度、小结和完成状态

## 8. 学生端学习流程（锁定版）

### 标准流程
进入章节  
→ 查看知识讲解  
→ 开始苏格拉底式问题链  
→ 提交文字思路 / 代码 / 任务结果  
→ 系统追问 / 提示 / 纠错  
→ 完成本节总结  
→ 提交结束

### 说明
- 学生不能直接无限制闲聊
- 系统默认围绕当前章节、当前知识点和当前任务引导
- 当学生明显不会时，系统按层级给提示
- 必要时给参考答案与反思总结，但不应一开始就给

## 9. 苏格拉底式对话策略（产品定义）

### 目标
不是“直接答题”，而是推动学生完成以下认知过程：
1. 识别问题
2. 回忆概念
3. 建立规则
4. 尝试作答
5. 发现错误
6. 修正思路
7. 总结迁移

### 分层策略
- 第 1 层：启发式追问
- 第 2 层：缩小范围
- 第 3 层：给局部提示
- 第 4 层：给半步答案
- 第 5 层：给完整答案 + 为什么

## 10. 教学资源自动生成策略

### 输入
- Word/PDF/PPT/TXT/MD/网页文本/图片

### 自动输出
1. 章节与小节切分
2. 知识点提取
3. 章节讲解初稿
4. 问题链初稿
5. 任务/练习初稿

### 原则
- 教师可编辑
- 生成内容必须可回写数据库
- 每次生成都保留版本
- 不允许只停留在模型回答层，不落库

## 11. 非功能需求

### 性能
- 一个班级同时在线使用（建议先按 40–60 人设计）
- 教师端课堂控制操作应有明显响应
- 单次模型响应速度不宜过长
- 代码运行须设置超时与资源限制

### 易用性
- 界面现代、清晰、卡片化
- 电脑端优先
- 学生操作路径短
- 教师端避免深层配置嵌套

### 安全性
- 账号密码加密存储
- API Key 不明文暴露给前端
- 文件上传类型与大小受控
- 代码执行沙箱隔离
- 模型输出受课堂安全策略约束
